Getting Started

Build your first application in Human — step by step.

0

Install

Human is distributed as a single Go binary. Install it with one command, then verify it's working.

terminal 
# Install the compiler
$ go install github.com/barun-bash/human/cmd/human@latest

# Verify it's installed
$ human version
Requirements

Go 1.21+ is required. If you don't have Go installed, get it from go.dev/dl. After installing, make sure $GOPATH/bin is in your PATH.

1

Your first app

Every Human project starts with a single line: a declaration of what you are building. This tells the compiler the name of your application and what kind of platform it targets. No imports, no configuration files, no boilerplate. Just one sentence.

app.human 
app TaskFlow is a web application
What this does

This single line declares an application called TaskFlow that targets the web. The compiler uses this to set up routing, project structure, and build configuration. You can also target mobile, desktop, or api platforms.

2

Define your data

Data declarations describe the entities in your application — the things your users create, read, update, and delete. Each field has a type, and you can add modifiers like unique, encrypted, or optional. Relationships between entities are expressed as plain English: belongs to, has many.

app.human 
data User:
  has a name which is text
  has an email which is unique email
  has a password which is encrypted text
  has a role which is either "user" or "admin"
  has an optional bio which is text
  has a created datetime
  has many Task

data Task:
  belongs to a User
  has a title which is text
  has a status which is either "pending" or "in_progress" or "done"
  has a priority which is either "low" or "medium" or "high"
  has a due date
  has many Tag through TaskTag
What this does

The compiler generates database tables, ORM models, TypeScript types, and migration files from these declarations. The unique modifier creates a database constraint. The encrypted modifier ensures the password is hashed at rest. Enum fields like status and role are type-safe — the compiler rejects any value not in the list. The belongs to / has many syntax creates foreign keys and join tables automatically.

3

Create pages

Pages describe what the user sees and how they interact with it. You write display statements, interaction handlers, input elements, and conditional states — all in English. The compiler turns these into fully functional frontend components with proper state management, event handling, and data fetching.

app.human 
page Dashboard:
  show a greeting with the user's name
  show a summary card with total tasks, completed tasks, and overdue tasks

  show a list of tasks sorted by due date
  each task shows its title, status, priority, and due date
  each task shows the status as a colored badge
  clicking a task opens a detail panel on the right

  there is a search bar that filters tasks by title
  there is a dropdown to filter by status

  if no tasks match, show "No tasks found. Create your first task!"
  while loading, show a skeleton screen

  there is a floating button to add a new task
  clicking the add button opens a form to create a Task
What this does

This generates a complete Dashboard page: a React component with a greeting header, summary statistics, a sortable and filterable task list, search input, status dropdown, an empty state message, a loading skeleton, and a floating action button that opens a creation form. Every interaction described in English becomes a real event handler. Every conditional becomes a proper render branch.

4

Build your API

API declarations define your backend endpoints. Each one specifies what it accepts, what preconditions it requires, what logic it runs, and what it returns. Validation, authentication, and error handling are expressed inline. No route files, no middleware configuration — just describe the behavior.

app.human 
api CreateTask:
  requires authentication
  accepts title, description, priority, and due date
  check that title is not empty
  check that title is less than 200 characters
  check that due date is in the future
  create a Task with the given fields and current user as owner
  set status to "pending" if not provided
  respond with the created task

api GetTasks:
  requires authentication
  fetch all tasks for the current user
  sort by due date
  support filtering by status
  support searching by title
  paginate with 20 per page
  respond with tasks and pagination info
What this does

The compiler generates Express (or your chosen framework) route handlers with input validation, authentication middleware, database queries, error responses, and pagination. Each check that statement becomes a validation rule that returns a clear error message on failure. Each fetch or create becomes a properly parameterized database operation. Filtering, sorting, and pagination are handled automatically.

5

Add authentication and security

Security in Human is not an afterthought. The authentication block configures your auth strategy, and policy blocks define role-based access control. The compiler enforces these rules across your entire application — if an API requires authentication but your security config is missing, the build fails with a helpful message.

app.human 
authentication:
  method JWT tokens that expire in 7 days
  method Google OAuth with redirect to /auth/google/callback
  passwords are hashed with bcrypt using 12 rounds
  all api requests require a valid token except SignUp and Login
  rate limit all endpoints to 100 requests per minute per user
  sanitize all text inputs against XSS
  enable CORS only for our frontend domain

policy FreeUser:
  can create up to 50 tasks per month
  can view only their own tasks
  can edit only their own tasks
  cannot delete completed tasks
  cannot export data

policy Admin:
  can view all users and their data
  can edit any task
  can delete any task
  can export data
  can view system analytics
What this does

The compiler generates JWT middleware, OAuth integration, bcrypt password hashing, rate limiting middleware, XSS sanitization, and CORS configuration. Policies become authorization guards applied to every relevant endpoint. The FreeUser policy creates usage limits enforced at the database level. The Admin policy grants elevated access. These are not just comments — they are compiled into real, tested access control code.

6

Set up workflows

Workflows describe what happens when events occur in your system. A user signs up, a task becomes overdue, a payment fails — each trigger starts a sequence of actions. These are compiled into event handlers, background jobs, and scheduled tasks with proper retry logic and error handling.

app.human 
when a user signs up:
  create their account
  assign FreeUser policy
  send welcome email with template "welcome"
  after 3 days, send email with template "getting-started"
  after 14 days, if they have fewer than 5 tasks,
    send email with template "need-help"

when a task is marked done:
  update the user's completed task count
  if all tasks for today are done,
    send congratulations notification
  log the completion for analytics
What this does

The signup workflow generates an event listener that runs after user creation: it assigns a default policy, sends an immediate welcome email, and schedules two delayed follow-up emails with conditional logic. The completion workflow updates aggregate counts, checks a condition across related records, and logs an analytics event. All timing, retries, and failure handling are managed by the compiled code.

7

Choose your stack and build

The build with block tells the compiler what frameworks and tools to target. Your entire application — data models, pages, APIs, security, workflows — compiles to the stack you choose. Change the target later and recompile; your logic stays the same.

app.human 
build with:
  frontend using React with TypeScript
  backend using Node with Express
  database using PostgreSQL
  deploy to Docker
What this does

The compiler reads your entire .human file and generates a complete project: React components with TypeScript types, Express API routes with middleware, PostgreSQL schemas with migrations, a Dockerfile, and a full test suite. Same source file could target Vue + FastAPI + MySQL — just change the build block.

Then run it from your terminal:

$ human check      # Validate your .human files
$ human build      # Compile to your target stack
$ human run        # Start the development server

Putting it all together

Here is a complete, minimal Human application combining every concept from the tutorial. This single file produces a full-stack, tested, deployable task manager.

app.human 
app TaskFlow is a web application

# ── Data ──

data User:
  has a name which is text
  has an email which is unique email
  has a password which is encrypted text
  has many Task

data Task:
  belongs to a User
  has a title which is text
  has a status which is either "pending" or "done"
  has a due date

# ── Pages ──

page Dashboard:
  show a list of tasks sorted by due date
  each task shows its title, status, and due date
  clicking a task toggles its status
  if no tasks match, show "No tasks yet."
  there is a form to create a Task

# ── API ──

api CreateTask:
  requires authentication
  accepts title and due date
  check that title is not empty
  create the task for the current user
  respond with the created task

# ── Security ──

authentication:
  method JWT tokens that expire in 7 days
  passwords are hashed with bcrypt

policy User:
  can view only their own tasks
  can create up to 100 tasks

# ── Workflows ──

when a user signs up:
  send welcome email with template "welcome"

# ── Build ──

build with:
  frontend using React with TypeScript
  backend using Node with Express
  database using PostgreSQL
